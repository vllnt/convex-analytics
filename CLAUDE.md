## Convex Component Pattern

- **Schema is truth** — `schema.ts` defines all tables and indexes. Every query/mutation references these.
- **Never edit `_generated/`** — these files are auto-generated by Convex codegen.
- **Component isolation** — `defineComponent("analytics")` creates a namespace. Child components (`aggregate`, `shardedCounter`, `rateLimiter`) are mounted via `component.use()`.
- **Consumer mounting** — users call `app.use(analytics)` in their `convex.config.ts`, then `new ConvexAnalytics(components.analytics)`.

## Key Patterns

- **Sharded counter** — 16-shard counter for high-throughput `track()` writes. `count()` reads are O(shards) not O(n).
- **Rate limiter** — token bucket per sessionId (100/min, burst 10). Silently drops excess events (returns null, no error).
- **Aggregate** — `DirectAggregate` with namespace `"eventName:YYYY-MM-DD"` for O(log n) date-range counts.
- **Session lifecycle** — auto-created on first event per sessionId. Closed by cron after 30min inactivity. `seqNum` derived from session's `eventCount` (TOCTOU-safe).
- **Daily rollups** — cron runs every 5min, aggregates last 10min of events into `daily_rollups` with pre-computed dimension breakdowns. Idempotent (uses `Math.max` merge).
- **Alias** — paginated reassignment (500/batch) of events+sessions from anonymousId to identifiedId, then user record merge.

## Common Pitfalls

1. **Don't `.collect()` the events table** — it can have millions of rows. Use indexes + `.take(limit)`.
2. **Event name validation** — must match `/^[a-zA-Z_][a-zA-Z0-9_]{0,63}$/`. Reject at track() and API layer.
3. **Property schema filtering** — when `event_schemas` entry exists, unknown properties are silently dropped, and type mismatches are silently dropped. This is intentional (not an error).
4. **Rollup idempotency** — uses `Math.max` not `+=` when merging into existing rollup rows. Safe to re-run.
5. **Counter drift** — `rebalance` cron detects >1% drift between sharded counter and actual event count. Log warning only.
